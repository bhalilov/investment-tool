2. Evidence Document Anatomy

Evidence docs are not database rows. They are compact, readable packets that preserve enough context for AI retrieval and reasoning.

Source ID
x_thread_2061424788300853554, original X URL, capture timestamp, canonical local JSON path.
Included Text
AJ root post, AJ replies, and only the user questions AJ directly answered when needed for meaning.
Excluded Text
Random user chatter, self-promo, duplicate thread copies, generated HTML/index markup.
Media
Screenshot local path, OCR text, vision summary, important numbers, source tweet id.
Hints
Ticker/topic/date/source hints for retrieval. These are routing clues, not final truth.
Linked Context
X thread links, Ghost links, and a note when the evidence is incomplete without following a link.
Evidence docs should make the truth searchable. They should not pretend to resolve every interpretation up front.

4. Component Responsibilities

Each component has a narrow job. The system avoids one giant prompt that tries to parse, classify, remember, and alert all at once.

Capture Engine

Does: pull X threads, media, Ghost content, and linked context.

Stores: raw API, canonical JSON, media files.

Does not: decide final investment meaning.

Evidence Builder

Does: dedupe, select AJ/direct-context posts, OCR screenshots, create RAG-ready docs.

Stores: evidence JSON/TXT.

Does not: upload raw noise to vector store.

Memory Maintainer

Does: revise ticker/topic notes as new evidence arrives.

Stores: live markdown notes and audit logs.

Does not: create endless versions.

RAG Exporter

Does: export clean evidence and memory notes to vector-store JSONL.

Stores: export files and upload ids.

Does not: treat vector store as the source of truth.

Dashboard Generator

Does: produce ticker timelines, graph views, daily pages, and investigation pages.

Stores: disposable views.

Does not: replace evidence or memory.

Chat Interface

Does: route questions to search, memory, evidence, and price tools, then cite sources.

Stores: optional chat transcripts.

Does not: answer from ungrounded model memory.

5. Chat And RAG Flow

Questions are answered by combining flexible retrieval with living ticker memory and exact source evidence.

User Question

Why did AJ change his mind about TSLA?

→

Question Router

Decides whether to use ticker memory, vector search, exact source lookup, or price data.

→

Retrieve Context

TSLA memory note, relevant evidence docs, Ghost/X links, screenshots.

→

Grounded Answer

Timeline, explanation, uncertainty, and clickable evidence links.

Bad Pattern

Dump every raw JSON, HTML index, and random reply into vector store, then ask the model to figure it out.

Good Pattern

Search clean evidence and living memory; fetch exact sources; synthesize with citations and uncertainty.

6. Generated Views

Views are useful but disposable. They should be regenerated from evidence and memory whenever the analysis improves.

Ticker Timeline

For PLTR, MU, TSLA: recent narrative, explicit actions, screenshots, open questions, and linked context.

Daily Brief

What changed today, especially for owned stocks and high-priority topics.

Graph View

Post to linked thread to Ghost article to stance change to later contradiction.

Investigation Page

Answer-specific pages for questions like: how long did AJ hold MU before trimming?

Source Browser

The old thread HTML idea, but treated as a source viewer rather than the intelligence layer.

Vector Export Status

Which evidence and memory documents have been exported or need refresh.

7. Final Mental Model

The system should preserve facts, build useful AI memory, and answer questions flexibly without pretending the world fits a rigid schema.

Do Not Build

  • A rigid database that needs a field for every possible future question.
  • A vector store full of raw duplicates and HTML sludge.
  • A one-call thread classifier that ignores ticker history.
  • Versioned memory markdown clutter.

Build Instead

  • Raw evidence archive as the truth layer.
  • Clean evidence docs for RAG.
  • Living ticker/topic memory notes.
  • Chat and dashboards that reason from evidence + memory.

Generated as static HTML/SVG-style infographics. No external libraries required.
